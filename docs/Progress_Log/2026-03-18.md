## 콤보 UI 뜨게 하기

### 추가 요청 대응 계획 (boss hit confirm only)
* 문제 원인은 `AttackState.StartComboStep()`가 공격 step 시작 시점에 바로 combo UI를 열고 있어, 실제 boss hit 여부와 상관없이 UI가 보인다는 점으로 정리했다.
* 수정은 combo UI의 show timing만 `DamageCaster`의 real hit confirm 경로로 옮기고, hide timing(`AttackState.Exit()`, death path, HUD init, `ShowHud(false)`)은 유지하는 방향으로 진행한다.
* `DamageCaster`에는 `didDamage == true`일 때만 1회 발행되는 hit-confirm event를 추가하고, `PlayerController`가 이 event를 받아 현재 pending combo step을 HUD에 표시하게 만든다.
* `AttackState`는 combo step 숫자를 직접 보여주지 않고, 현재 단계만 `PlayerController`에 pending 값으로 넘기도록 바꾼다.
* 검증 기준은 `miss -> hidden 유지`, `boss hit -> combo UI 표시`, `next hit -> 숫자 갱신`, `dash cancel/hit/stun/death/end -> hide`다.

### 오늘 반영한 작업
* `Assets/Scripts/Common/Combat/DamageCaster.cs`에 `OnAttackHitConfirmed` event를 추가했다. 같은 공격 윈도우 안에서는 첫 번째 `didDamage == true` 순간에만 1회 발행되도록 막았다.
* `Assets/Scripts/Player/PlayerController.cs`가 combo step을 `pending` 상태로 들고 있다가, `DamageCaster.OnAttackHitConfirmed`를 받는 순간에만 `ShowComboHud(step)`를 호출하도록 바꿨다.
* `Assets/Scripts/Player/States/AttackState.cs`는 공격 step 시작 시 combo UI를 직접 열지 않고, 현재 step 숫자만 `SetPendingComboHudStep(step)`로 준비하도록 수정했다.
* 기존 hide 경로(`AttackState.Exit()`, death path, HUD init, `ShowHud(false)`)는 유지해 stale combo UI 정리 동작은 그대로 보존했다.
* `Canvas.prefab` 안의 existing `Text_Combo` / `Image_Combo` 구조를 그대로 재사용해, scene hierarchy 재구성 없이 code gate만으로 visibility를 처리했다.

* `Canvas.prefab` 안에 필요한 노드가 이미 있으므로, prefab 구조를 다시 바꾸지 않는 쪽이 scene reference drift를 줄인다.
